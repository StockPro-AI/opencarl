<task type="checkpoint:decision" gate="blocking">
  <decision>Should output format markers be rebranded to OpenCARL or retained for backward compatibility?</decision>
  <context>
    The following output format markers currently use CARL naming:
    
    1. **XML tags in injector.ts** (lines 143, 226):
       - `<carl-rules>` and `</carl-rules>` wrap injected rules in system prompts
    
    2. **HTML comment markers in agents-writer.ts** (lines 3-6):
       - `<!-- CARL-START - DO NOT EDIT -->`
       - `<!-- CARL-END - DO NOT EDIT -->`
       - These mark the OpenCARL section in AGENTS.md files
    
    Rebranding these markers would provide complete consistency but may affect existing AGENTS.md files that use the old markers. Retaining them preserves backward compatibility.
  </context>
  <options>
    <option id="option-a">
      <name>Complete rebranding</name>
      <pros>
        - Full consistency with OpenCARL branding
        - SOURCE-04 fully satisfied
        - No legacy CARL references remain
      </pros>
      <cons>
        - Existing AGENTS.md files with CARL-START markers won't be auto-detected
        - Users would need to manually remove old integration or run remove command first
      </cons>
      <implementation>
        - Change `<carl-rules>` to `<opencarl-rules>` in injector.ts
        - Change CARL-START/CARL-END to OPENCARL-START/OPENCARL-END in agents-writer.ts
      </implementation>
    </option>
    <option id="option-b">
      <name>Retain for backward compatibility</name>
      <pros>
        - Existing AGENTS.md integrations continue to work
        - No user action required for upgrade
        - Safe transition path
      </pros>
      <cons>
        - SOURCE-04 only partially satisfied (documented as intentional)
        - Legacy CARL references remain in output format
      </cons>
      <implementation>
        - Add code comment explaining intentional retention for backward compatibility
        - Document in VERIFICATION.md that this is an intentional exception to SOURCE-04
      </implementation>
    </option>
  </options>
  <resume-signal>Select: option-a (complete rebranding) or option-b (retain for compatibility)</resume-signal>
</task>
